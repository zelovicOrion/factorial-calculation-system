# Distributed Factorial Calculation System (Multithreaded Implementation)

## Overview
This project implements a **distributed factorial calculation system** consisting of two interacting applications:  

1. **Client Module** – responsible for validation and request routing.  
2. **Server Module** – responsible for computation and persistence.  

The modules communicate via **HTTP (REST)** protocol.

---

## 1. Functional Requirements

### Module A – Interface / Gateway
- **Entry Point:** Accepts calculation requests via `HTTP POST`.  
- **Validation:** Ensures input parameters are valid:  
  - Target number > 0  
  - Thread count > 0  
- **Logic:** Formats and forwards requests to the Computation Module based on event type.  
- **Identification:** Each calculation is assigned a unique string identifier generated by the executor (server) side.

### Module B – Computation / Execution Engine
Handles the following **events**:

1. **Start:** Initializes the calculation with the specified number of threads.  
2. **Stop:** Interrupts an active calculation by its unique identifier.  
3. **Status:** Returns the current state of a calculation:  
   - `Running` / `Stopped` / `Completed`  
4. **Result:** Returns the final factorial value (supports extremely large numbers).  
5. **Recommence:** Resumes a calculation interrupted due to system restart or shutdown.

---

## 2. Technological Constraints

- **Multithreading:**  
  - Use `ForkJoinPool` for task decomposition.  
  - The number of parallel subtasks is determined by the request parameter.  

- **Synchronization:**  
  - Use a **Semaphore** to solve the Reader-Writer problem for shared calculation status data.  

- **File I/O:**  
  - Persist intermediate states using **NIO.2** (`Paths`, `Channels`, or File operations).  

- **Database Interaction:**  
  - Use `JdbcTemplate` for relational database operations.  

- **Configuration:**  
  - The persistence method (File or Database) must be selectable via an external configuration file.  
  - Switching storage should not require modifications to core logic.

---

## 3. Communication
- All interactions between modules use **RESTful HTTP endpoints**.  
- Client module forwards validated requests; server module handles execution and persistence.
